# 自己回復調整型 自動ベットシステム構想

## 目的

`live_trigger` の最終目標は、前日候補抽出から当日監視、承認、実ベット、結果報告、異常時の停止と復旧までを一体で扱える、自己回復調整型の自動ベット基盤を作ることです。

この構想では、ロジック本体は決定論的に動かし、エージェントは状況説明、承認導線、異常検知、運用支援を担当します。

## 理想の運用イメージ

外出先からでも、次の流れで運用できる状態を目指します。

1. ユーザーが LINE などから「今どうなってる」と問い合わせる
2. エージェントが現在の監視状況、次の対象レース、異常有無、待機中か実行中かを返す
3. 締切 10 分前から 5 分前の監視ウィンドウで最終確認を行う
4. ベットサイズとロジック内容を最終確認し、必要なら承認依頼を送る
5. 異常値なら自動停止し、承認なしでは進めない
6. 承認されたものだけ実ベットする
7. 実行結果を即時報告する
8. 必要なら遠隔で状況確認、停止、再開、金額調整、profile 停止を行う

## システムの基本原則

- ベット内容の生成はロジックが行う
- エージェントは勝手に買い目を変更しない
- 実ベットは承認済みデータだけを処理する
- 異常検知時は自動停止を既定動作にする
- セッション異常、締切超過、金額異常は必ずログに残す
- 状況確認、承認、実行、結果報告は全て DB に証跡を残す

## 必要な機能ブロック

### 1. Trigger Layer

前日候補抽出、当日監視、締切 10 分前から 5 分前の最終判定を担います。

対象:

- `app.py`
- `boxes/`
- `watchlists/`
- `ready/`
- `auto_system/app/modules/01_sync_watchlists.py`
- `auto_system/app/modules/02_evaluate_targets.py`

### 2. Intent Layer

レースごとの最終候補を、実行可能なベット単位へ展開します。

対象:

- `bet_intents`
- ベットサイズ計算
- profile ごとの金額制御
- 承認待ちへ進めるための request 生成

### 3. Approval Layer

承認依頼を作り、承認済みか却下か、期限切れかを管理します。

必要な役割:

- 承認依頼の作成
- 承認期限の管理
- approve / reject / pause / resume / amount change の受け付け
- 承認ログ保存

### 4. Notification Layer

LINE など外部ツールへ状況通知と承認依頼を送る層です。

通知内容:

- 現在のシステム状態
- 次の監視対象
- 承認待ちベット
- 異常停止
- 実行結果
- 日次サマリー

### 5. Agent Layer

ユーザーからの問い合わせに応答し、DB とログを読んで現状を要約します。

期待する役割:

- 「今どうなってる」への即答
- 異常の要約
- 承認待ち内容の説明
- 遠隔からの運用コマンド受け付け
- 必要時の修正提案

### 6. Execution Layer

Teleboat へ実際に接続し、`air / assist_real / armed_real` を実行します。

対象:

- `auto_system/app/core/teleboat.py`
- `auto_system/app/modules/03_execute_air_bets.py`
- `bet_executions`
- `session_events`

### 7. Reconcile Layer

実行後の契約番号、実行時刻、照合結果、結果報告を残します。

必要な役割:

- 契約番号保存
- 実行スクリーンショット保存
- 実行結果と照合結果の突合
- 日次損益の集計

### 8. Self-Healing Layer

異常時に止まるだけでなく、止まり方と復旧方法を管理します。

想定する自己回復:

- UI は生きているが auto loop が止まったときの再起動
- Teleboat session 切れ時の再ログイン誘導
- 監視対象が空のときの watchlist 再同期
- 一時的な取得失敗時の再試行
- DB ロックや I/O 異常の待機再実行
- 異常が続く場合の自動停止と通知

## 理想の 1 レースフロー

1. 前日に `watchlist` を作成する
2. 当日に `target_races` へ同期する
3. 締切 10 分前に `monitoring` に入る
4. `beforeinfo` を見て最終判定する
5. `GO` なら `bet_intents` を作る
6. エージェントが承認依頼を送る
7. 異常があれば `stop`、問題なければ承認待ち
8. 承認後に `assist_real` または `armed_real` を実行する
9. 契約番号と実行時刻を保存する
10. 結果を報告する
11. 必要なら次レースの条件や金額を調整する

## 追加したい DB テーブル

- `approval_requests`
  - 承認待ち、承認済み、却下、期限切れを管理する
- `operator_commands`
  - pause / resume / stop / amount change / profile disable を受ける
- `notification_logs`
  - LINE などへ何を送ったか記録する
- `system_snapshots`
  - 定期的な状態要約を保存する
- `anomaly_events`
  - 異常値、セッション異常、締切超過、復旧試行を残す

## 遠隔運用で返せるようにしたい情報

最低限、次をエージェントが即答できる状態を目指します。

- システムが起動中か停止中か
- 今の `execution_mode`
- 次に監視へ入るレースの時刻
- `imported / monitoring / intent_created / pending_approval / executed / expired` の件数
- 直近の異常
- 直近の実行結果
- 今日の Air Bet 件数
- 今日の実ベット件数

## 承認フローの方針

- 新しいロジックは必ず `air` から始める
- 安定後に `assist_real` へ上げる
- 実機確認後に `armed_real` へ上げる
- `armed_real` は既定で承認必須にする
- 金額変更は上限付きにする
- 締切 5 分未満は新規実行禁止にする

## 異常時の既定動作

- セッション切れ: 実行停止、再ログイン要求、通知
- 承認期限切れ: 実行しない、expired として通知
- 金額異常: stop
- レース番号不整合: stop
- 締切超過: 実行しない
- 通知失敗: DB に残し、別チャネルへ再送を試みる

## 段階導入のロードマップ

### Phase 1

- 現行の `air / assist_real / armed_real` を安定運用する
- ステータス表示とエラーログを整える
- 日次サマリーを作る

### Phase 2

- 「今どうなってる」に答える Status Agent を作る
- `system_snapshots` を追加する
- LINE などへの状態通知を始める

### Phase 3

- `approval_requests` を導入する
- 承認依頼、却下、期限切れを扱えるようにする
- 遠隔コマンドで pause / resume / stop / amount change を受けられるようにする

### Phase 4

- `armed_real` を承認フローに完全接続する
- 結果報告を自動化する
- 異常時の自動停止と復旧試行を入れる

### Phase 5

- 自己回復調整型の運用へ進める
- 状況要約、異常検知、運用提案、承認フローをエージェントが支援する
- 人は最終承認と例外処理に集中できる形にする

## この構想のゴール

最終的には、`live_trigger` を次のようなシステムに育てることを目標とします。

- 普段は自動で監視する
- 必要な場面だけ人に承認依頼を送る
- 外出先からでも状況確認できる
- 異常時は自動で止まり、理由を説明できる
- 復旧可能なものは自己回復を試みる
- その日の運用結果を最後まで報告できる
